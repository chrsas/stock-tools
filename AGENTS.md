# 协作规范

1. 标题和列表适度使用，避免过度分层。
2. 减少使用表情符号。
3. 禁止使用“不是……而是……”句式。
4. 避免使用破折号。
5. 搜索资料时尽量减少使用中文网站信源。
6. 不要用日文
7. Git 提交信息使用中文，简洁描述本次改动目的。
8. 开工前通读 `tasks.md` 第 0 节与第 6 节，再读当前要做的阶段一节；改动采集、状态机、数据库或适配器前先读 `docs/architecture.md`（技术栈、数据模型、状态机细节）；已完成阶段的详细规范在 `docs/phases-archive.md`，仅供追溯，日常开发无需阅读。每完成一个阶段，将其详细规范移入归档，`tasks.md` 原位置只留一行交付摘要。
9. 提交前运行 `.\scripts\check_quality.ps1`，确保 lint、格式检查、类型检查和自动化测试全部通过。
10. 提交前检查 `git status` 和差异内容，确保凭据、本地配置、运行数据和原始采集文件未进入提交。

# 代码开发原则

1. 单一职责按模块切分，一个文件只对应一个变更理由。文件过大时拆成包，`__init__.py` 重新导出公开符号作为兼容垫片，保持既有导入路径不变。参照 `kol_archive/archive`、`kol_archive/presentation`、`kol_archive/web` 的拆法。
2. 解析层写成纯函数，无副作用，输入原始负载、输出规范化模型。网络、浏览器、数据库等 IO 留在边界，平台解析见 `kol_archive/adapters`。
3. 依赖注入。`Archive` 等接收外部传入的 `connection` 与 `settings`，不在内部自行创建，便于测试注入内存库与桩件。
4. 用类型表达状态，优先 `Enum` 与 `dataclass`，避免裸字符串与可表示的非法状态。状态判断用 `is` 比较枚举。
5. 存证遵循追加不可篡改。版本与事件表只追加，原文与 OCR 原样保留，派生表可由原始数据重算。
6. 平台差异收敛在适配器（防腐层）。接新平台时新增适配器文件，核心存证逻辑不改。重装存证只限雪球。
7. 重构搬迁函数时保留兼容导入。若被搬迁函数被测试以字符串路径 `monkeypatch.setattr("module.fn")` 打桩，同步更新桩点路径到新模块，并让调用方以模块限定方式调用以保证打桩生效。
8. 前端样式集中在全局 `frontend/src/style.css`，组件内不写 `<style>`。抽取视图为子组件时逐字搬运模板，渲染 DOM 保持一致，UI 视觉冻结。
